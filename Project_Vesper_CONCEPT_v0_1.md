# Project Vesper 戦闘企画書 v0.1

## 1. 企画概要

**Project Vesper** は、Godot 4.7 / GDScript で開発する、2.5D見下ろし視点のスタイリッシュアクション戦闘プロトタイプである。  
現時点の目標は、最終的なローグライク／ハクスラ／TPS化まで一気に作ることではなく、まずは**強敵1体〜少数精鋭との戦闘が面白いか**を検証することである。

現在のプロジェクト方針でも、正式な3Dモデルやアニメーションより、カプセルやボックスなどの仮モデルで素早くテストプレイできることが優先されている。  
また、開発方針としても、GodotのNode / Scene構成に合わせ、GDScriptで責務を分けて実装することが前提になっている。

---

## 2. 最初に目指すもの

最初に目指すのは、以下のような戦闘体験である。

```text
敵の攻撃予兆を見て、
回避・パリィ・差し込み攻撃を選び、
敵のリズムを読み切って主導権を奪い、
最後にスタイリッシュな一撃を叩き込む。
```

重要なのは、**音楽に同期するリズムゲームではない**こと。  
Project VesperのリズムはBPMではなく、**敵の構え・攻撃予兆・間合い・攻撃テンポ**から生まれる。

目指す感触は、

```text
音楽に合わせるゲームではなく、
敵の殺意のリズムを読み、捌き、崩すゲーム。
```

である。

---

## 3. 戦闘コンセプト

### 基本コンセプト

**Project Vesperの戦闘は、反射神経だけで殴り合うゲームではなく、敵の構え・間合い・テンポを読んで、美しく主導権を奪うゲームにする。**

現在のREADMEでも、敵の攻撃予兆を見て避ける、後隙を狙って反撃する、スタミナを管理する、将来的にパリィやジャスト回避で高リターンを得る、という方向性が明記されている。  
この方針をさらに発展させ、戦闘の核を以下のように定義する。

```text
速い攻撃は捌く。
遅い攻撃は差し込む。
掴みは避ける。
アーマー技は空振らせる。
フェイントは待つ。
読み切ったらFlowを奪い、スタイリッシュな一撃で締める。
```

---

## 4. 差別化方針

Project Vesperは、既存のアクションゲームと以下のように差別化する。

| 比較対象 | 主な特徴 | Project Vesperの方向 |
|---|---|---|
| Hades系 | 部屋制ローグライト、軽快な戦闘 | まずは強敵戦の密度を重視 |
| Hi-Fi Rush系 | 音楽BPMに同期したリズムアクション | 音楽ではなく敵攻撃のリズムを読む |
| Souls系 | 重めの回避・後隙取り | よりスタイリッシュでFlow重視 |
| 格ゲー | 人間同士の見えない読み合い | PvE向けに、見える予兆と判断の読み合いへ翻訳 |
| ハクスラ | 装備・数値・ドロップの楽しさ | 初期段階では操作判断と戦闘感触を優先 |

Project Vesperの独自性は、

```text
敵のリズムを覚えるゲームではなく、
敵のリズムを読んで崩し、主導権を奪うゲーム
```

に置く。

## 5. やらないこと

初期段階では、以下はやらない。

```text
音楽BPMとの同期
見えない二択
大量の雑魚を倒すハクスラ戦闘
ランダムステージ生成
装備ドロップ
永続成長
複数キャラ実装
正式3Dモデル前提の処理
大規模な状態管理フレームワーク
過剰な抽象化
```

開発ガイドラインでも、当面はランダムステージ生成、装備ドロップ、永続成長、複数キャラ、正式モデル前提の処理、大規模な状態管理などは禁止方針になっている。  
ここは守る。欲張ると、まだ肉を焼いている途中なのにフルコースの皿を並べることになる。見た目だけ忙しい。

## 6. 戦闘の基本ループ

戦闘は以下の流れで成立させる。

```text
敵が攻撃予兆を出す
↓
プレイヤーが技の種類を判断する
↓
回避 / パリィ / 差し込み / 距離取りを選ぶ
↓
成功・失敗を判定する
↓
成功時はFlow上昇、敵の隙、ヒットストップ、カメラ演出を発生
↓
Flowが高まったらスタイリッシュな一撃を狙える
↓
次の攻防へ
```

このループの狙いは、単なる暗記ではない。

敵の攻撃パターンを覚えることは重要だが、毎回、

```text
今は安全に回避するか？
パリィでFlowを稼ぐか？
遅い攻撃に差し込むか？
欲張って強攻撃を入れるか？
```

という判断が残るようにする。

---

## 7. PvE向けの読み合い設計

格ゲーのような読み合いは取り入れるが、**見えない二択は避ける**。

PvPなら見えない択も心理戦として成立するが、PvEでは理不尽に感じやすい。  
そのため、Project Vesperでは以下の原則を守る。

```text
見えない二択ではなく、見える選択肢。
ただし、見えてから最適行動を選ぶには少し練習がいる。
```

敵の攻撃には、必ず見分けられる予兆を用意する。

| 敵の予兆 | 意味 | 推奨対応 |
|---|---|---|
| 武器を小さく引く | 速い斬り | パリィ、回避 |
| 武器を大きく後ろに引く | 遅延重攻撃 | 差し込み、距離取り |
| 片手を前に出す | 掴み | 回避、バックステップ |
| 体が赤く光る | アーマー技 | 殴らず回避、後隙を取る |
| 構えたまま止まる | フェイント気味 | 早押しせず待つ |

プレイヤーが負けたときに、

```text
今のは見えていたのに焦った
パリィに頼りすぎた
差し込める技だったのに逃げた
アーマー技に殴りかかってしまった
```

と思えることが理想である。

```text
ランダムで死んだ
CPUがズルした
見えなかった
```

と思わせてはいけない。

## 8. アクションのスピード感

Project Vesperでは、攻撃フレームを全体的に長めにする。

目指すのは、

```text
素早いけど軽くない。
派手だけど雑ではない。
見て判断できるが、完璧に捌くには練習がいる。
```

というスピード感である。

目安は以下。

| 技タイプ | 予兆時間の目安 | 目的 |
|---|---:|---|
| 速い斬り | 0.35〜0.5秒 | 緊張感を出すが、理不尽にはしない |
| 通常攻撃 | 0.5〜0.8秒 | 見て防御行動を選べる |
| 遅延重攻撃 | 0.9〜1.3秒 | 差し込み判断を促す |
| 掴み | 0.6〜0.9秒 | パリィ不可を見て避ける |
| 大技 | 1.2〜2.0秒 | 位置取り、差し込み、特殊反撃のチャンス |

速すぎるゲームにはしない。  
敵の構えを見て「これは捌く」「これは潰す」「これは逃げる」と判断できる余地を残す。

---

## 9. プレイヤー行動の役割

プレイヤーの基本行動には、それぞれ明確な役割を持たせる。

| 行動 | 役割 | 強い場面 | 弱い場面 |
|---|---|---|---|
| 回避 | 安全寄りの防御 | 大技、掴み、範囲外への離脱 | 追尾、連続攻撃 |
| ジャスト回避 | 中リスク中リターン | 敵の攻撃を紙一重で避ける | タイミング失敗時 |
| パリィ | 高リターン防御 | 武器攻撃、単発攻撃 | 掴み、フェイント、遅延 |
| 通常攻撃 | 差し込み | 遅い攻撃、詠唱、溜め | 速い攻撃、アーマー技 |
| 強攻撃 | 高火力パニッシュ | 大きな後隙、ダウン中 | 発生前に潰される |
| 待ち | フェイント対策 | 遅延・フェイント確認 | 掴み、距離詰め |

重要なのは、どれか1つを万能にしないこと。

```text
回避だけで勝てる
パリィだけで勝てる
通常攻撃連打で勝てる
```

となると浅くなる。  
行動ごとに「勝てる相手」と「負ける相手」を作る。

---

## 10. 敵攻撃の基本タイプ

最初に実装する敵攻撃は、複雑にしすぎない。  
まずは以下の4種類を核にする。

### 1. 速い斬り

```text
発生が速い
ダメージは低め
差し込みは難しい
パリィ・回避で対応する
```

役割：プレイヤーの雑な攻撃や油断を止める。

### 2. 遅延重斬り

```text
予兆が長い
ダメージは高い
早押しパリィを狩る
発生前に通常攻撃で差し込める
```

役割：プレイヤーに「待つか、差し込むか」を判断させる。

### 3. 掴み

```text
パリィ不可
ガード不可
近距離で使う
回避やバックステップで対応する
```

役割：パリィ偏重を咎める。

### 4. アーマー叩きつけ

```text
発生中に軽攻撃では止まらない
大ダメージ
予兆はかなり分かりやすい
回避後に大きな後隙を殴れる
```

役割：雑な差し込みを咎める。

## 11. Flowシステム案

将来的に、成功行動を評価するための仮称として **Flow** を導入する。

Flowは、敵の攻撃をうまく処理したときに上昇する。

| 行動 | Flow上昇 |
|---|---:|
| 通常回避成功 | 小 |
| ジャスト回避成功 | 中 |
| パリィ成功 | 中〜大 |
| 遅い攻撃への差し込み成功 | 中 |
| 大技の後隙に強攻撃成功 | 大 |
| 被弾 | 減少 |
| 空振り・無駄行動 | 少し減少 |

Flowが一定以上になったら、特殊反撃やスタイリッシュな一撃を出せる。

```text
敵の大技を読み切る
↓
回避・パリィ・差し込みで主導権を奪う
↓
Flowが溜まる
↓
強攻撃や専用入力でスタイリッシュカウンター
```

ただし、初期実装ではFlowゲージを急いで作らなくてよい。  
まずはパリィ、ジャスト回避、差し込み成功時にログや簡易表示を出す程度でもよい。

---

## 12. 成功時の気持ちよさ

Project Vesperでは、成功時の手応えを非常に重視する。

現在の実装優先順位でも、ヒットストップ、カメラ揺れ、簡易VFX、パリィ、コンボ表示が次の価値として挙がっている。

成功時には、以下を段階的に追加する。

```text
ヒットストップ
短いスロー
カメラ揺れ
カメラ寄り
効果音
簡易VFX
敵の怯み
Flow表示
スタイリッシュカウンター演出
```

特に、パリィ成功や大技への差し込み成功は、通常ヒットより明確に気持ちよくする。

## 13. 初期実装の優先順位

Codexに実装させる場合、まずは以下の順番で進める。

### Step 1：プレイヤー攻撃の状態管理

現在は、プレイヤー攻撃が入力直後に判定発生する制限がある。  
まずは攻撃を以下の状態に分ける。

```text
予備動作
攻撃判定発生
後隙
```

通常攻撃と強攻撃で、それぞれ発生・持続・後隙を調整できるようにする。

### Step 2：敵攻撃の予兆表示

敵の攻撃前に分かりやすい予兆を出す。

```text
速い斬り
遅延重斬り
掴み
アーマー叩きつけ
```

まずは色・スケール・向きガイド・簡易ラベルなど、仮表示でよい。

### Step 3：敵攻撃タイプの実装

敵の攻撃を1種類ではなく、役割の違う複数タイプにする。

```text
速い攻撃：パリィ・回避向き
遅い攻撃：差し込み向き
掴み：パリィ不可、回避向き
アーマー技：差し込み不可、回避後に反撃
```

### Step 4：パリィ

パリィを実装する。

```text
短い受付時間
武器攻撃に成功
掴みには失敗
成功時は敵を止める
成功時にヒットストップ
```

### Step 5：差し込み判定

敵の遅い攻撃や溜め攻撃中に、プレイヤーの通常攻撃が当たったら敵の攻撃を中断できるようにする。

ただし、アーマー技は中断不可にする。

### Step 6：ヒットストップ・カメラ揺れ・簡易VFX

成功時の気持ちよさを追加する。  
ここは地味に見えて重要。ここがないと、設計は正しくても手触りが死ぬ。

### Step 7：Flowの簡易表示

最初はゲージでなくてもよい。

```text
Flow +1
Perfect Dodge
Parry
Interrupt
```

のようなテキスト表示だけでも十分。

## 14. Codex向け実装ルール

Codexに実装させる場合、以下を守らせる。

```text
- Godot 4.7 / GDScript を前提にする
- 既存の Node / Scene 構成を尊重する
- Unity版の逐語移植はしない
- gameplay scripts は責務ごとに小さく分ける
- 調整値は可能な限り @export 変数にする
- 正式モデルや正式アニメーション前提にしない
- プリミティブ表示、色変更、簡易VFXでよい
- まずテストプレイしやすさを優先する
- 大規模なイベントバスや過剰な抽象化は入れない
- 旧Unity版ファイルは明示指示なしに編集・削除しない
```

これは既存の開発ガイドラインとも一致する。

## 15. この企画の一文要約

Project Vesperは、

```text
敵の攻撃リズムを読み、回避・パリィ・差し込みで主導権を奪い、Flowを高めてスタイリッシュな一撃を決める、PvE向け2.5Dデュエルアクション。
```

である。

まずはここをブレさせない。  
ローグライト化、ハクスラ化、TPS化は後で考える。今は**戦闘の芯**を焼く段階である。
